# Drawcast 기본 Rubric (VLM 채점)

채점자: Codex CLI(GPT-5.4, vision). 입력으로 **원 질문과 렌더된 PNG만** 받는다. scene JSON은 의도적으로 주어지지 않는다 — 이 rubric은 사람이 화면에서 보는 것과 동일한 신호로만 품질을 판정한다(구조·레이블의 기계적 확인은 별도의 JSON 룰 베이스 트랙 `runner/metrics.ts`가 담당하므로 여기서 JSON을 참조하면 두 트랙의 신호가 섞여 오염된다).
응답은 반드시 아래 **JSON 포맷**으로만 출력한다. 자연어 설명은 `notes` 필드에 담는다.

## 5개 채점 축

### 1. structure (0~5)
질문이 요구하는 주요 엔티티(노드)와 관계(엣지)가 모두 존재하는지.

| 점수 | 기준 |
|---|---|
| 5 | 모든 필수 요소 + 에러 분기·경계 케이스 등 의미 있는 추가 요소가 명확히 표현됨 |
| 4 | 질문이 명시한 모든 필수 노드·엣지가 누락 없이 존재 |
| 3 | 필수 노드·엣지 중 1개가 빠졌거나, 불필요한 노드가 3개 이상 추가되어 핵심 구조가 흐려짐 |
| 2 | 필수 노드·엣지 중 2~3개 누락, 또는 전체 구조의 절반 이하만 표현됨 |
| 1 | 핵심 구조를 대부분 표현하지 못함. 노드 수가 기대의 절반 이하 |
| 0 | 질문이 요구한 구조와 완전히 다르거나, 빈 다이어그램 |

### 2. labels (0~5)
각 노드·엣지 레이블이 **의미적으로 정확하고 구체적**인지. (한국어·영어 혼용 OK)

| 점수 | 기준 |
|---|---|
| 5 | 모든 레이블이 도메인 용어를 정확히 사용하고 간결함. 오타·오역 없음 |
| 4 | 대부분 정확. "검증" 대신 "확인" 수준의 동의어 치환이 1~2곳 있으나 의미 손실 없음 |
| 3 | 레이블이 지나치게 일반적("처리", "단계", "박스" 등)이거나, 질문에 명시된 고유 명사가 3개 이상 누락 |
| 2 | 레이블 절반 이상이 비어 있거나 의미와 무관한 문자열 |
| 1 | 대부분 레이블 없음, 또는 영어/한국어가 뒤섞여 일관성 없음 |
| 0 | 레이블 자체가 없거나 판독 불가 |

### 3. layout (0~3)
겹침 없이, 일관된 흐름 방향(위→아래 / 좌→우 등)으로, 시각적으로 따라가기 쉬운지.

| 점수 | 기준 |
|---|---|
| 3 | 노드 간 겹침 없음, 흐름 방향 일관(예: 플로우차트는 위→아래), 엣지가 노드를 가로지르지 않음 |
| 2 | 대체로 읽힘. 정렬이 일부 어긋나거나 엣지 교차가 1~2곳 있으나 흐름 파악 가능 |
| 1 | 노드 겹침이 3곳 이상이거나, 흐름 방향이 섞여 어느 쪽으로 읽어야 할지 불명확 |
| 0 | 공간 구조가 엉켜 다이어그램 해석 불가 |

### 4. readability (0~3)
글자 크기·색 대비·시각적 클러터 관점에서 한눈에 읽히는지.

| 점수 | 기준 |
|---|---|
| 3 | 모든 레이블이 노드 경계 안에 들어오고, 배경 대비 충분, 여백 균형 |
| 2 | 대체로 읽힘. 레이블 1~2개가 박스 밖으로 살짝 나오거나 대비가 약함 |
| 1 | 레이블이 3개 이상 잘리거나 겹침. 노드 크기가 너무 작아 텍스트 판독 어려움 |
| 0 | 상당 부분 읽을 수 없음(글자가 서로 겹치거나 배경과 구분 불가) |

### 5. intent_fit (0~5)
원 질문의 **의도와 다이어그램 유형**이 결과와 정확히 매치되는지.

| 점수 | 기준 |
|---|---|
| 5 | 질문이 요구한 다이어그램 유형(플로우차트/시퀀스/ERD/상태머신 등)이 정확하고, 질문 맥락의 핵심 디테일(실패 분기, 루프, 보상 트랜잭션 등)까지 표현 |
| 4 | 다이어그램 유형 정확. 핵심 디테일 중 1개 사소하게 누락 |
| 3 | 다이어그램 유형은 맞으나, 질문이 명시적으로 요구한 핵심 요소(예: "실패 시 보상 흐름")가 빠져 있음 |
| 2 | 다이어그램 유형이 부정확(예: 시퀀스 요청인데 플로우차트로 그림)하거나 질문 의도의 절반 이하만 반영 |
| 1 | 다이어그램 형식이 질문과 다르고, 내용도 표면적으로만 관련 |
| 0 | 질문과 무관한 다이어그램 |

## 응답 포맷 (필수)

```json
{
  "structure": 4,
  "labels": 5,
  "layout": 3,
  "readability": 2,
  "intent_fit": 4,
  "major_issues": ["결제 실패 분기 누락"],
  "minor_issues": ["노드 3-4가 약간 겹침"],
  "notes": "전체적으로 의도는 잘 반영했으나 결제 실패 처리가 빠져 실무 품질에는 못 미침."
}
```

추가 규칙:
- 점수는 정수만 사용 (소수점 금지)
- `major_issues`는 1점 이상 감점 사유, `minor_issues`는 감점 없는 지적
- `notes`는 3줄 이내 한국어
- JSON 외의 텍스트를 출력하면 채점 실패로 간주

## 가중치와 정규화

최종 점수:

```
weighted = Σ(weight_i × score_i)
max      = Σ(weight_i × max_score_i)   # max_score: structure=5, labels=5, layout=3, readability=3, intent_fit=5
total    = weighted / max              # 0.0 ~ 1.0
```

질문의 `rubric_weights`가 축을 생략하면 **1.0**로 간주한다.

## 판정 기준 (집계용)

- **pass**: `total >= 0.80` **그리고** 5점 스케일 축(structure·labels·intent_fit)이 모두 **3점 이상**, 3점 스케일 축(layout·readability)이 모두 **2점 이상**
- **fail**: `total < 0.55` **또는** 5점 스케일 축 중 하나라도 **2점 이하** **또는** 3점 스케일 축 중 하나라도 **1점 이하**
- **borderline**: 그 외

> **판정 의도**: 단순 총점 합격이 아니라 모든 축에서 최소 수준을 넘어야 pass다. 레이아웃이 엉망이거나 레이블이 부정확하면 총점과 무관하게 fail 처리한다.

집계 리포트에서 `pass_rate = pass_count / total_runs`로 표시한다.
